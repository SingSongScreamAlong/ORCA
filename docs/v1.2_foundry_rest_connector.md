# v1.2 — Foundry REST Connector (read-only)

## Purpose

v1.1 shipped connection *scaffolding* — a config shape, a read-only client abstraction, a
mock client, and a secret-free health check — but the only client that returned data was the
mock; the "real" client was an honest placeholder. v1.2 fills that gap with a **real,
read-only REST connector** (`RestFoundryClient`) that talks to a Foundry tenant's documented
v2 platform API over `httpx`. It is the working path behind the same abstraction and the same
health check.

> **What is real, and what is still unverified.** The connector is a genuine implementation:
> it performs the OAuth2 token exchange, carries a bearer token, calls Foundry's v2 ontology
> endpoints, parses responses, and is exercised by tests against an injected mock transport.
> What it has **not** done is run against the live ORCA tenant — that first real call belongs
> to an operator with credentials, because CI/sandbox has no outbound network and no secrets.
> Exact scope names and endpoint paths should be confirmed against your tenant's API docs.

## What this is — and is not

- **Is:** a real read-only REST client (httpx) implementing the v1.1 `FoundryClient`
  surface; OAuth2 client-credentials **or** pre-issued bearer-token auth; calls to Foundry's
  documented v2 ontology/object endpoints; the default `client_kind` when Foundry is enabled.
- **Is not:** any write path, full sync, AIP execution, or background data movement. It reads
  ontology metadata and (optionally) a single named demo object. No records are read by the
  health check.

## Safety properties (unchanged from v1.1, plus)

- **Read-only.** Only `GET` requests are issued, plus the OAuth2 token `POST`. There is no
  method that writes to Foundry.
- **Disabled by default** (`ORCA_FOUNDRY_ENABLED=false`); **no credentials** needed for dev or
  CI; **no secrets committed**.
- **Secrets never appear in errors or logs.** The token request carries the client secret, so
  a rejected exchange is reported only as `OAuth2 token request was rejected (HTTP <status>)`
  — the request body is never echoed. Read failures report only `GET <path> returned HTTP
  <status>`. Tests assert the client secret and token never appear in any error string.
- **No live tenant in tests.** Every test injects an `httpx.MockTransport`; nothing contacts a
  real Foundry tenant. The first live call is a deliberate, operator-run manual test.

## How auth works

`RestFoundryClient._bearer_token()`:

1. If `ORCA_FOUNDRY_TOKEN` is set, use it directly (no exchange).
2. Otherwise `POST {tenant}/multipass/api/oauth2/token` with
   `grant_type=client_credentials`, `client_id`, `client_secret`, and (optionally)
   `scope=ORCA_FOUNDRY_SCOPES`; cache and reuse the returned `access_token`.

Every read sends `Authorization: Bearer <token>`.

## Read-only endpoints (Foundry v2 API)

| Method | Foundry path |
| ------ | ------------ |
| `health_check()` | `GET /api/v2/ontologies` (metadata listing; reads no records) |
| `get_object_type_metadata(type)` | `GET /api/v2/ontologies/{ontology}/objectTypes/{type}` |
| `get_object_by_id(type, id)` | `GET /api/v2/ontologies/{ontology}/objects/{type}/{id}` |
| `list_demo_objects(type, limit)` | `GET /api/v2/ontologies/{ontology}/objects/{type}?pageSize={limit}` |

Paths follow Foundry's public v2 API; verify them and the required scopes against your
tenant's API docs (see [`foundry_connection_setup.md`](foundry_connection_setup.md)).

## Configuration (new in v1.2)

| Variable | Meaning |
| -------- | ------- |
| `ORCA_FOUNDRY_CLIENT` | `rest` (default — the httpx REST connector) or `sdk` (the v1.1 placeholder). |
| `ORCA_FOUNDRY_SCOPES` | Optional space-separated OAuth2 scopes for the client-credentials grant (e.g. an ontology-read scope). Blank uses the client's default granted scopes. |

All other `ORCA_FOUNDRY_*` variables are as in v1.1. `httpx` is now a **runtime** dependency
(it was dev-only before), because the connector needs it in production.

## Client selection

`build_foundry_client(config)`:

- enabled + `client_kind == "rest"` (default) → **`RestFoundryClient`** (this milestone);
- enabled + `client_kind == "sdk"` → `RealFoundryClient` (v1.1 placeholder, fails gracefully);
- disabled → `MockFoundryClient` (so dev/CI work with no credentials).

The health endpoint/CLI is unchanged: `GET /api/v1/integrations/foundry/health` (admin-only)
and `python -m app.foundry.health`. When enabled with the REST connector, `ok: true` now means
the connector actually reached the tenant and listed ontologies.

## Tests

`tests/backend/test_foundry_rest_client.py` (10 tests, no network) prove: the
client-credentials exchange happens then the read carries the issued bearer token; a
pre-issued token skips the exchange; the four endpoint paths are shaped correctly; the listing
uses `pageSize`; HTTP read errors and token rejections raise secret-free
`FoundryConnectionError`s; and the factory returns the REST client by default and the SDK
placeholder when selected. The v1.1 connection tests (mock/placeholder/health) and all earlier
suites continue to pass.

## Manual test against a real tenant

See [`foundry_connection_setup.md`](foundry_connection_setup.md). In short: in a local
(gitignored) `.env`, set `ORCA_FOUNDRY_ENABLED=true`, `ORCA_FOUNDRY_TENANT_URL`,
`ORCA_FOUNDRY_ONTOLOGY_API_NAME`, an auth method, and (optionally) `ORCA_FOUNDRY_SCOPES`, then
run `python -m app.foundry.health`. A healthy tenant returns `ok: true` with an
`ontology_count`. A scope/auth problem returns a clear, secret-free `ok: false`.

## Open questions for production

- The exact OAuth2 scope name(s) the tenant requires for ontology/object reads.
- Confirmation of the v2 endpoint paths for this tenant (and the published ontology API name).
- Token lifetime/refresh policy (this connector caches a token for the client's lifetime; a
  long-lived process should add refresh-on-expiry).
- How ORCA's per-case need-to-know maps onto Foundry's security model when real object reads
  are wired into case workflows (today reads are confined to the manual health check).
