# v0.4 — Auth/RBAC + Workspace Hardening

v0.4 makes the analyst workflow enforced by the system, not just by UI convention. It
adds real authenticated identities, six roles, capability-based authorization on every
endpoint, separation of duties on approvals (with an explicit admin override), and case
membership/assignment. No scraping, dark-web collection, autonomous hunting, face
search, offender/victim targeting, external OSINT integrations, or Palantir sync.

## Authentication

Identities are first-class. For local/dev use, the caller selects a seeded user with the
`X-ORCA-User` header (the frontend's dev user switcher sets it via a cookie); when
absent, the configured default user is assumed. An unknown user gets **401**. Production
replaces this single resolver with real credential verification (e.g. OIDC) — the rest
of the authorization stack is unchanged.

## Roles

Six roles, least privilege:

| Role                    | Purpose                                                        |
| ----------------------- | -------------------------------------------------------------- |
| `admin`                 | Full access; may override the self-review control.             |
| `case_manager`          | Create cases, assign users, manage case status; publish reports.|
| `analyst`               | Propose observations, evidence, relationships; draft reports.  |
| `reviewer`              | Approve/reject/needs_more_review/quarantine; may also propose. |
| `viewer`                | Read case material; cannot create or decide.                   |
| `partner_export_viewer` | Access **only** published (approved) report packages.          |

report packages.

## Separation of duties

* A user **cannot decide on their own proposed intelligence** (observation or evidence).
  This is enforced server-side using the proposer recorded on the item.
* An **admin override** (`override: true`, admin-only) bypasses the self-review block and
  is recorded as a **distinct audit event** (`review.override` / `evidence.override`).
* A reviewer cannot wield the override — attempting it returns **403**.

## Audit

Every privileged action writes to the append-only audit log: case creation and member
assignment, observation intake, evidence create/link/decide/verify, relationship
creation, review decisions, report generation and publishing, and every admin override
(as its own event).

## Endpoints added

| Method & path                  | Purpose                                              |
| ------------------------------ | ---------------------------------------------------- |
| `GET  /me`                     | The authenticated principal + its capabilities.      |
| `GET  /users`                  | Demo user roster (dev switcher).                     |
| `GET  /cases/{id}/members`     | Case members.                                        |
| `POST /cases/{id}/members`     | Assign a user (case manager / admin).                |
| `POST /reports/{id}/publish`   | Publish a draft as an approved package.              |
| `GET  /reports/published`      | List published report packages.                      |
| `GET  /reports/{id}`           | Get a report (partners: published only).             |

All existing endpoints now carry capability guards; permission failures return **403**.

## Walkthrough — analyst proposes → reviewer approves → viewer reads → partner sees approved report only

```bash
B=http://localhost:8000/api/v1
hdr() { echo "X-ORCA-User: $1"; }

CASE=$(curl -s $B/cases -H "$(hdr casey)" -H 'content-type: application/json' \
  -d '{"title":"RBAC demo","owner":"casey"}' | jq -r .id)        # case manager creates
SRC=$(curl -s "$B/sources" -H "$(hdr ana)" | jq -r '.[0].id')

# analyst proposes (and cannot approve)
E=$(curl -s $B/entities -H "$(hdr ana)" -H 'content-type: application/json' \
  -d '{"entity_type":"advertisement","value":"ad-rbac"}' | jq -r .id)
OBS=$(curl -s $B/observations -H "$(hdr ana)" -H 'content-type: application/json' -d "{
  \"case_id\":\"$CASE\",\"timestamp\":\"2026-01-01T00:00:00Z\",\"source_id\":\"$SRC\",
  \"collector\":\"ana\",\"notes\":\"ad-rbac\",\"confidence\":0.7,\"entity_ids\":[\"$E\"]}" | jq -r .id)
ITEM=$(curl -s "$B/review?case_id=$CASE" -H "$(hdr rae)" | jq -r ".[]|select(.subject_id==\"$OBS\")|.id")
curl -s -o /dev/null -w "analyst approve -> %{http_code}\n" -X POST $B/review/$ITEM/decision \
  -H "$(hdr ana)" -H 'content-type: application/json' -d '{"decision":"approve"}'   # 403

# reviewer approves another user's proposal
curl -s $B/review/$ITEM/decision -H "$(hdr rae)" -H 'content-type: application/json' \
  -d '{"decision":"approve"}' | jq .status                                          # approved

# analyst drafts a report; case manager publishes it
REP=$(curl -s -X POST $B/cases/$CASE/report -H "$(hdr ana)" | jq -r .id)
curl -s -X POST $B/reports/$REP/publish -H "$(hdr casey)" | jq .status              # final

# viewer reads case material; partner cannot, but sees the published package
curl -s -o /dev/null -w "viewer /cases -> %{http_code}\n"  $B/cases -H "$(hdr vic)"      # 200
curl -s -o /dev/null -w "partner /cases -> %{http_code}\n" $B/cases -H "$(hdr partner)"  # 403
curl -s -o /dev/null -w "partner /evidence -> %{http_code}\n" $B/evidence -H "$(hdr partner)" # 403
curl -s $B/reports/published -H "$(hdr partner)" | jq '.[].title'                   # the package
```

## Frontend

- **Dev user switcher** in the top bar (sets the `orca_user` cookie); current user and
  role are shown.
- Controls are **hidden or disabled** when the role lacks the capability (New case,
  Add observation/evidence, review decisions, evidence decisions, report generate/publish,
  member assignment).
- **Role-aware Review Queue and Evidence Locker** (decisions only for reviewers/admins;
  admins get an automatic override when blocked by self-review).
- Clear **permission error states** (403 reads render a calm "you don't have access"
  notice rather than an outage).
- **Members** tab on Case Detail for assignment (case managers / admins).

## Verified guarantees (tests)

`tests/backend/test_rbac.py` proves: analyst can propose but not approve; reviewer can
approve another's proposal; reviewer cannot approve their own without override; viewer
cannot mutate; partner cannot browse raw material (only published reports); permission
failures return 403 (unknown users 401); privileged actions are audited; admin override
writes a distinct override event.
