# v1.0 — AIP-assisted Analyst Copilot (propose-only)

## Purpose

v1.0 adds a **local, provider-agnostic AI assistance layer** that helps analysts work
faster while preserving ORCA's core rule: **AI proposes, analysts decide.** AI output
never becomes authoritative case material automatically — every result is a *proposed*
suggestion, draft, candidate, or advisory note that requires human review.

This is **not** a live Palantir AIP integration. It is designed to map cleanly onto
Palantir AIP later (same propose-only contract), but it runs entirely locally with a
deterministic mock provider and **no external credentials**.

## Core principle

> AI output must never become authoritative case material automatically. Every AI result
> is a proposed object, candidate, draft, or advisory note that requires human review.

Every result carries explicit metadata: `generated_by_ai: true`, `provider`,
`generated_at`, `source_material_ids`, `status: "proposed"`, `requires_human_review: true`.
The Copilot has **no write path** into case material — an analyst must act on a suggestion
through the normal, audited intake / review / relationship / report flows.

## Allowed AI use cases

1. Summarize **approved** case material.
2. Extract candidate entities from analyst notes or approved observations (proposed only).
3. Suggest possible duplicate entities.
4. Suggest relationship candidates from approved observations (proposed only).
5. Draft report language from **approved observations only**.
6. Generate timeline summaries from approved case events.
7. Flag missing citations.
8. Flag unsupported claims in draft reports.
9. Suggest caveats for report-package export.
10. Identify review gaps (e.g. evidence without verification, relationships without
    approved support).

## Forbidden AI use cases

No autonomous collection · no scraping · no dark-web browsing · no face recognition or
matching · no victim/offender targeting · no identity claims without human review · no
direct-contact suggestions · no law-enforcement conclusion generation · no risk scoring of
real people · no automated report approval · no AI claims in partner exports unless
reviewed and approved. None of these is implemented, and the Copilot's only data source is
approved case material the caller is authorized to read.

## Endpoints

All under `/api/v1`, case-membership gated (`require_case_material_read`):

| Method & path | Purpose |
| ------------- | ------- |
| `POST /cases/{id}/ai/summarize` | Summary of approved material + review-gap/caveat notes |
| `POST /cases/{id}/ai/extract-entities` | Candidate entities (proposed; flags possible duplicates) |
| `POST /cases/{id}/ai/suggest-relationships` | Candidate `appears_with` links from co-occurrence |
| `POST /cases/{id}/ai/draft-report-section` | Draft section from approved observations |
| `POST /cases/{id}/ai/check-citations` | Missing-citation + unsupported-claim flags on a draft |
| `POST /cases/{id}/ai/timeline-summary` | Summary of the approved timeline |

All return a single `AiAssistResult` envelope (summary, suggestions, proposed_entities,
proposed_relationships, report_draft, citation_gaps, unsupported_claims, notes, meta).

## Access control

- **Case membership gated.** Only members who may read case material reach the Copilot;
  the endpoints use the same need-to-know guard as the rest of the case surface.
- **Partner export viewers are blocked.** They lack `read_case_material`, so every Copilot
  endpoint returns 403 — partners get approved report packages only, never AI assistance.
- **Unassigned users** get the generic 403 that does not reveal case existence.
- **Approved material only.** The Copilot's context is built from approved observations,
  approved relationships (with approved support), and approved evidence — proposed,
  rejected, needs-more-review, and quarantined material is excluded.

## Audit

Every Copilot request writes an append-only audit event (`ai_assist.summarize`,
`ai_assist.extract_entities`, `ai_assist.suggest_relationships`,
`ai_assist.draft_report_section`, `ai_assist.check_citations`, `ai_assist.timeline_summary`)
recording the actor, case, provider, and result counts.

## Human-review requirement

AI output is advisory. To become case material it must go through the existing audited
workflow — an analyst records an observation/entity/relationship (which enters the review
queue as *proposed*), or edits a report draft. The Copilot never approves, never asserts a
finding about a person, never bypasses review, and is never autonomous. Results are not
persisted as authoritative objects; they are returned with `status: "proposed"` and
`requires_human_review: true`.

## Provider abstraction

`backend/app/ai_assist/provider.py` defines an `AiProvider` protocol and a deterministic,
offline `MockProvider` (the default, `ORCA_AI_PROVIDER=mock`). The mock derives suggestions
from approved material with simple, reproducible rules — no randomness, no network, no
credentials — so tests are deterministic and the Copilot is usable out of the box. A real
provider (e.g. Palantir AIP) would implement the same protocol and the same propose-only
contract; none is included, and none is required.

## Mapping to Palantir AIP

This realizes the **propose-only AIP** plan from
[`v0.9_palantir_foundry_mapping.md`](v0.9_palantir_foundry_mapping.md) §11: AIP may
*suggest* (candidate relationships, candidate entity merges, summaries of approved
material, review-queue triage), and each suggestion enters the human review queue. The
`AiProvider` seam is where an AIP-backed provider would slot in, preserving every boundary
documented here.

## Frontend

A calm **Copilot** tab on Case Detail with a prominent banner — *"AI suggestions are
proposed only. Human review is required."* — and actions for the six endpoints. Results
render as clearly-labelled proposals (never facts), with a "proposed · requires human
review" badge and provider/timestamp provenance. The panel never writes case material; it
offers copy-to-draft and dismiss. It is hidden from anyone who cannot read case material
(partner export viewers never reach it).

## Walkthrough (curl)

```bash
B=http://localhost:8000/api/v1
hdr() { echo "X-ORCA-User: $1"; }
CASE=$(curl -s $B/cases -H "$(hdr admin)" | jq -r '.[0].id')

curl -s -X POST $B/cases/$CASE/ai/summarize -H "$(hdr ana)" -d '{}' \
  -H 'content-type: application/json' | jq '{summary, meta}'
curl -s -X POST $B/cases/$CASE/ai/check-citations -H "$(hdr ana)" -H 'content-type: application/json' \
  -d '{"draft_text":"The suspect is guilty. Evidence shows a link."}' | jq '.unsupported_claims'
# partner is refused
curl -s -o /dev/null -w "partner -> %{http_code}\n" -X POST $B/cases/$CASE/ai/summarize \
  -H "$(hdr partner)" -H 'content-type: application/json' -d '{}'   # 403
```

## Verified guarantees (tests)

`tests/backend/test_ai_copilot.py` proves: summaries and drafts use approved material only;
entity extraction and relationship suggestions are proposed-only and create nothing; the
citation checker flags unsupported claims and missing citations; every result carries
`generated_by_ai` + `requires_human_review`; partner export viewers cannot access the
Copilot and unassigned users get 403; requests are audited; and the mock provider is
deterministic. The v0.2–v0.9 suites continue to pass.

## Boundaries & remaining risks

The Copilot adds no capability ORCA lacks and no autonomy. Remaining risks are the usual
ones for assistive AI and are mitigated by design: over-trust of suggestions (mitigated by
the proposed-only contract, the persistent banner, and no write path); prompt content in a
future real provider (mitigated by approved-material-only context and the `AiProvider`
seam, which is where redaction/governance would live); and determinism is guaranteed only
for the mock provider (a real provider would require its own evaluation).
