# v0.5 — Relationship Graph & Discovery

v0.5 turns ORCA's confirmed relationships into a navigable graph: neighbourhoods, case
subgraphs, and shortest paths between entities. This is the "discover relationships"
half of the mission made visible. It is read-only and additive — no scraping, no
collection, no autonomous hunting, no external integrations.

## Principle: discovery only over approved relationships

Every graph query reads the **authoritative record of approved relationships**. Proposed,
rejected, and needs-more-review relationships never appear in discovery — you can only
traverse links a human has approved. Discovery is RBAC-gated (`read_case_material`), so
`partner_export_viewer` cannot reach it.

> The Neo4j projection (`app/repositories/graph_repository.py`) mirrors the same edges
> when `ORCA_GRAPH_ENABLED=true`, as a scale/visualization accelerator. These queries
> read the relational record, so they are correct and testable on any backend.

## Endpoints (under `/api/v1`, all `read_case_material`)

| Method & path                       | Purpose                                          |
| ----------------------------------- | ------------------------------------------------ |
| `GET /cases/{id}/graph`             | A case's relationship subgraph (nodes + edges).  |
| `GET /graph/neighbors/{entity_id}`  | An entity's one-hop neighbourhood.               |
| `GET /graph/path?source=&target=`   | Shortest path between two entities (BFS).         |

Shapes:

```jsonc
// GraphView
{ "nodes": [{ "id", "entity_type", "value" }],
  "edges": [{ "id", "source_entity_id", "target_entity_id", "relationship_type", "confidence" }] }

// PathView
{ "found": true, "length": 2, "nodes": [ ...ordered... ], "edges": [ ...ordered... ] }
```

## Walkthrough (curl)

```bash
B=http://localhost:8000/api/v1
H='X-ORCA-User: admin'
CASE=$(curl -s $B/cases -H "$H" | jq -r '.[0].id')

# The seed case: two advertisements linked by a shared phone number.
curl -s "$B/cases/$CASE/graph" -H "$H" | jq '{nodes: (.nodes|length), edges: (.edges|length)}'

# One entity's neighbourhood
NID=$(curl -s "$B/cases/$CASE/graph" -H "$H" | jq -r '.nodes[0].id')
curl -s "$B/graph/neighbors/$NID" -H "$H" | jq '.edges[].relationship_type'

# Shortest path between two entities
A=$(curl -s "$B/cases/$CASE/graph" -H "$H" | jq -r '.edges[0].source_entity_id')
Z=$(curl -s "$B/cases/$CASE/graph" -H "$H" | jq -r '.edges[0].target_entity_id')
curl -s "$B/graph/path?source=$A&target=$Z" -H "$H" | jq '{found, length}'
```

## Frontend

A **Graph** tab on Case Detail renders the case subgraph as a calm, dependency-free SVG
node-link diagram (nodes placed on a circle, edges labelled with the relationship type).
It is server-rendered and deterministic — no client graph library.

## Verified guarantees (tests)

`tests/backend/test_graph.py` proves: the case subgraph contains the approved edges;
neighbours are correct; shortest path is found (and reports absence); discovery excludes
non-approved relationships; and the endpoints are RBAC-gated (partner → 403).
