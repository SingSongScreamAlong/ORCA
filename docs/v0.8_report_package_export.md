# v0.8 — Report Package Export

v0.8 adds **partner-ready report package export** on top of the approved-only report
draft (v0.2), the evidence locker (v0.3), file upload (v0.7), hash verification, RBAC
(v0.4), and case membership (v0.6). An assigned analyst/reviewer generates an immutable
export snapshot of a case's **approved** material: a Markdown report plus a JSON evidence
manifest, with content hashes. A partner export viewer can view and download the package
for assigned cases — and nothing else.

This milestone is **export/report packaging only**. No scraping, dark-web collection,
autonomous hunting, face search, offender/victim targeting, external OSINT integrations,
Palantir sync, AI automation, bulk social monitoring, or direct-contact workflows.

## What a package contains

A package is built from **approved material only** and never bundles raw evidence files —
evidence is represented by metadata and SHA-256 hashes.

**Markdown report** (`report.md`): case title, summary; generated time and generated-by;
report status; handling level; export caveats; approved observations; approved
relationships; cited evidence metadata with hash values and verification status; a
timeline summary; and an explicit exclusions/caveats section.

**JSON manifest** (`manifest.json`): package/case identity, generated time/by, handling
level, caveats, counts, and an `evidence` array. Each evidence entry carries:
`evidence_id`, `case_id`, `observation_id`, `title`, `evidence_type`,
`original_filename`, `mime_type`, `size_bytes`, `sha256`, `status`, `source_id`,
`source_name`, `captured_at`, `captured_by`, `verification`, `handling_notes`, and
`legal_flags`.

An optional **ZIP** (`report-package-{id}.zip`) bundles both files.

## Exclusions

By construction, a package **excludes**:

- proposed, rejected, needs-more-review, and quarantined observations and evidence;
- raw evidence file bytes (metadata + hashes only);
- the case audit log and the relationship graph (never part of partner exports).

## Endpoints

| Method & path                                  | Purpose                                            |
| ---------------------------------------------- | -------------------------------------------------- |
| `POST /cases/{id}/report/package`              | Generate a package (approved material only).       |
| `GET  /report-packages`                        | List packages for the caller's cases.              |
| `GET  /report-packages/{id}`                   | Package metadata.                                  |
| `GET  /report-packages/{id}/report`            | Download the Markdown report.                      |
| `GET  /report-packages/{id}/manifest`          | Download the JSON evidence manifest.               |
| `GET  /report-packages/{id}/package`           | Download the ZIP (report + manifest).              |

## Permissions

| Action            | admin | case_manager | analyst | reviewer | viewer | partner_export_viewer | unassigned |
| ----------------- | :---: | :----------: | :-----: | :------: | :----: | :-------------------: | :--------: |
| Generate package  |   ✓   |      ✓       |    ✓    |          |        |                       |            |
| List / view / download | ✓ |      ✓       |    ✓    |    ✓     |   ✓    |          ✓            |            |

Generation requires the `generate_report` capability **and** active mutating membership
in the case. Listing/viewing/downloading requires `view_approved_reports` **and** active
membership (administrators see all). A `partner_export_viewer` can therefore view and
export the package for assigned cases, but `GET /cases/{id}/evidence`, `/graph`,
`/audit`, and `/observations` all require `read_case_material`, which partners lack —
so they never reach raw material. An unassigned user (no membership) gets a generic 403
that does not reveal whether the case or package exists.

## Integrity & determinism

The package records `report_sha256` and `manifest_sha256`. The manifest is serialized
canonically (`json.dumps(..., indent=2, sort_keys=True)` over JSON-native values), so a
downloaded manifest always re-hashes to the recorded `manifest_sha256`; likewise the
report. ZIP entries use a fixed timestamp so identical content yields a byte-stable
archive. Each manifest evidence entry includes a `verification` field computed at
generation by re-hashing the stored bytes (`verified` / `mismatch` /
`unverified_no_bytes` / `no_hash`).

## Audit

| Action                            | When                                  |
| --------------------------------- | ------------------------------------- |
| `report_package.generated`        | A package is generated.               |
| `report_package.report_downloaded`| The Markdown report is downloaded.    |
| `report_package.manifest_downloaded` | The manifest is downloaded.        |
| `report_package.downloaded`       | The ZIP package is downloaded.        |

Denied attempts are not recorded as separate audit rows (the request rolls back before
commit), consistent with v0.7; the denial is still enforced and successful exports are
audited.

## Frontend

- An **Export** tab on Case Detail lists the case's packages with metadata (status,
  handling level, counts, content hashes, caveats) and **Download Report / Manifest /
  Package** actions; authorised roles get a **Generate report package** action.
- The **Reports** page surfaces report packages for every role that may view approved
  reports — the partner export viewer's window — with the same download actions and no
  access to raw case material.
- Calm, evidence-first styling; clear caveats and 403 states.

## Walkthrough (curl)

```bash
B=http://localhost:8000/api/v1
hdr() { echo "X-ORCA-User: $1"; }
CASE=$(curl -s $B/cases -H "$(hdr admin)" | jq -r '.[0].id')

# an analyst generates a package from approved material
PKG=$(curl -s -X POST $B/cases/$CASE/report/package -H "$(hdr ana)" | jq -r .id)

# a partner (assigned) lists and downloads it — report, manifest, and zip
curl -s $B/report-packages -H "$(hdr partner)" | jq '.[].title'
curl -s $B/report-packages/$PKG/report   -H "$(hdr partner)" | head -5
curl -s $B/report-packages/$PKG/manifest -H "$(hdr partner)" | jq '.counts'
curl -s -o pkg.zip -w "zip %{http_code}\n" $B/report-packages/$PKG/package -H "$(hdr partner)"

# a partner cannot reach raw material, and cannot generate
curl -s -o /dev/null -w "partner evidence -> %{http_code}\n" $B/cases/$CASE/evidence -H "$(hdr partner)"   # 403
curl -s -o /dev/null -w "partner generate -> %{http_code}\n" -X POST $B/cases/$CASE/report/package -H "$(hdr partner)"  # 403
```

## Verified guarantees (tests)

`tests/backend/test_report_package.py` proves: a package includes approved observations
only; proposed/rejected observations and rejected/quarantined evidence are excluded;
approved evidence under approved observations is cited; the manifest carries SHA-256 +
source metadata and recorded hashes match the downloads; the ZIP contains the report and
manifest; a partner can view/download the package but not raw evidence/graph/audit and
cannot generate; an unassigned user is refused; and generation and downloads are audited.
The PostgreSQL integration test generates a package (JSONB manifest), downloads all three
artifacts, and confirms the recorded hashes. The v0.2–v0.7 suites continue to pass.
