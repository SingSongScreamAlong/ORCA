# v1.3 — Foundry Read Endpoints (admin-only, read-only)

## Purpose

v1.2 shipped a verified read-only Foundry **connector** (and CLIs: `health`, `discover`).
v1.3 surfaces those reads **inside the ORCA API** so they can drive an admin screen, a demo,
or (later) a guarded import — without anyone running a Python one-liner. It is the smallest
"read-into-ORCA" step: admin-only diagnostic endpoints that proxy the connector's read-only
methods. No write path; no case data is created.

## Endpoints

All under `/api/v1/integrations/foundry`, **admin-only**, **read-only**:

| Endpoint | Returns |
| -------- | ------- |
| `GET /health` | Connection health (v1.1). |
| `GET /discover` | Ontologies the credential can see, plus object types (when an ontology is configured, or always in mock mode). |
| `GET /object-types/{object_type}` | Object-type metadata. |
| `GET /objects/{object_type}?limit=N` | A small sample of objects (`limit` 1–50, default 10). |
| `GET /objects/{object_type}/{object_id}` | A single object by id. |

Every response carries a **`mode`** field (`mock` | `real`) so the data source is
unambiguous.

## Behaviour & safety

- **Admin-only.** Non-admins get a generic `403` (case_manager, analyst, reviewer, viewer,
  partner_export_viewer, and unassigned users alike).
- **Read-only.** Each endpoint maps to a connector `GET`; nothing writes to Foundry.
- **Mock by default.** With Foundry disabled (dev/CI default), the deterministic mock client
  answers (`mode: "mock"`) — so the endpoints, and any UI built on them, work with no tenant
  or credentials. Enable Foundry (`ORCA_FOUNDRY_ENABLED=true` + a token/credentials) to get
  live tenant reads (`mode: "real"`).
- **Secret-free errors.** Connector failures are translated to clean HTTP responses —
  `400` for a configuration problem (e.g. ontology not set for object reads), `502` for an
  upstream Foundry/network error — carrying only the connector's already secret-free message.
- **Not case-audited.** Like the health check, these are admin/connection diagnostics, not
  case actions, so they are not written to the case audit log. (A guarded *import* of Foundry
  objects into ORCA records — a future milestone — would be audited and analyst-reviewed.)

## Tests

`tests/backend/test_foundry_read_endpoints.py` (25 cases) exercise every endpoint in mock
mode (no network): admin happy paths, `mode: "mock"`, the `limit` bounds (`422` out of
range), and admin-only `403`s across all non-admin roles. The connector's real-mode request
shaping and secret-safety are covered by `test_foundry_rest_client.py`.

## Not in scope (yet)

A UI panel over these endpoints, and a **guarded import / sync-in** that maps Foundry objects
to ORCA records as *proposed, analyst-reviewed* items (read-only from Foundry, case-scoped,
no raw evidence, audited, provenance = Foundry RID). Both build on this endpoint surface.
