# v0.6 — Case Membership & Authorization Scoping

v0.6 adds **need-to-know** access control. v0.4 RBAC answers *what kind of user are
you?*; v0.6 case membership answers *which case are you allowed to touch, and how?*. A
non-administrator must hold an **active membership** in a case to see or act on it, and
the membership's **case role** decides what they may do there. Administrators access and
manage every case.

This is read-and-write authorization only — no scraping, dark-web collection, autonomous
hunting, face search, offender/victim targeting, external OSINT integrations, Palantir
sync, AI automation, bulk social monitoring, or direct-contact workflows. ORCA stays a
lawful, auditable, analyst-controlled workspace.

## Global role vs. case role

| Concept | Question it answers | Where it lives | Example |
| ------- | ------------------- | -------------- | ------- |
| **Global role** (v0.4) | What kind of user are you? | `users.role` | `analyst` |
| **Case role** (v0.6)   | What may you do *in this case*? | `case_members.case_role` | `viewer` on case A, `analyst` on case B |

A user holds **one** global role but may be assigned a **different** case role per case.
Effective permission is the intersection of three checks:

1. **Global capability** — the v0.4 RBAC capability for the action (e.g. `review_decide`).
2. **Active membership** — an `active` row in `case_members` for this user and case
   (administrators bypass this).
3. **Case role** — the membership's case role permits the action (e.g. only `reviewer`
   may review; only `case_manager` may manage the roster).

A global reviewer who is only a *viewer* on case A cannot approve anything in case A. A
global analyst with no membership in case A cannot even see that case A exists.

## Case roles and membership status

Case roles: `case_manager`, `analyst`, `reviewer`, `viewer`, `partner_export_viewer`.
Membership status: `active`, `inactive`, `revoked` — only `active` grants access.

When a user is assigned without an explicit case role, it defaults from their global
role. The user who **creates** a case is automatically enrolled as its `case_manager`,
so they can work it and enrol others immediately.

means no active membership. ✓ = allowed, blank = denied.

The `partner_export_viewer` reaches **only** approved (published) report packages, and
only for cases they are assigned to. They never browse raw observations, evidence, the
graph, the audit log, or unapproved material. An `unassigned` user can do nothing with a
case — including learning whether it exists.

## No leakage on denial

An unauthorised access to a case returns a **generic 403** —
`{"detail": "You do not have access to this case."}` — that reveals nothing: not the
case title, summary, counts, sources, evidence metadata, or even whether the case
exists. A non-member receives the **same** 403 for a real case id and a random one, so
the API cannot be used to enumerate cases. (Global capability denials — e.g. a partner
hitting `/cases` — still return the v0.4 role-based 403, which leaks no case specifics.)

## Membership lifecycle

| Method & path                                   | Purpose                                         |
| ----------------------------------------------- | ----------------------------------------------- |
| `GET    /cases/{id}/members`                    | The case roster (any active member).            |
| `POST   /cases/{id}/members`                    | Assign a user (admin or the case's manager).    |
| `PATCH  /cases/{id}/members/{membership_id}`    | Change a member's case role and/or status.      |
| `DELETE /cases/{id}/members/{membership_id}`    | Revoke a member's access (sets status `revoked`).|

There is **one** membership row per (case, user). Re-adding a removed member reactivates
that same row, so there is never more than one — let alone more than one *active* —
membership for a user in a case.

## Audit behaviour

Every membership change writes an append-only audit event with the actor, the case, the
membership id, the target user, the target case role, the resulting status, and the prior
value where relevant:

| Action                       | When                                   |
| ---------------------------- | -------------------------------------- |
| `case.member_added`          | A new membership is created.           |
| `case.member_reactivated`    | A removed membership is restored.      |
| `case.member_role_changed`   | A member's case role changes.          |
| `case.member_status_changed` | A member's status changes via `PATCH`. |
| `case.member_deactivated`    | A member is revoked via `DELETE`.      |

These join the v0.2–v0.5 audit events (intake, review decisions, evidence
create/link/decide/verify, relationship creation, report generation/publishing, admin
overrides). The audit log remains append-only.

## Seeded demo memberships

The demo case enrols one member per role — `casey` (case_manager), `ana` (analyst),
`rae` (reviewer), `vic` (viewer), and `partner` (partner_export_viewer). A seventh user,
`nomad` (global analyst), is deliberately assigned to **no** case, so the unassigned-user
path can be demonstrated. The in-memory backend seeds these automatically; the full
analyst loop still works out of the box.

## Walkthrough (curl)

```bash
B=http://localhost:8000/api/v1
hdr() { echo "X-ORCA-User: $1"; }
CASE=$(curl -s $B/cases -H "$(hdr admin)" | jq -r '.[0].id')   # the demo case

# 1. admin opens everything; the unassigned user sees no cases at all
curl -s $B/cases -H "$(hdr admin)" | jq 'length'                       # >= 1
curl -s $B/cases -H "$(hdr nomad)" | jq '.'                            # []

# 2. admin assigns the four working roles (already seeded here; shown for completeness)
curl -s -X POST $B/cases/$CASE/members -H "$(hdr admin)" -H 'content-type: application/json' \
  -d '{"username":"ana","case_role":"analyst"}'                        # 201 or 422 if present

# 3. an analyst member proposes; 4. a reviewer member approves
SRC=$(curl -s $B/sources -H "$(hdr ana)" | jq -r '.[0].id')
E=$(curl -s $B/entities -H "$(hdr ana)" -H 'content-type: application/json' \
  -d '{"entity_type":"alias","value":"walk-1"}' | jq -r .id)
OBS=$(curl -s $B/observations -H "$(hdr ana)" -H 'content-type: application/json' -d "{
  \"case_id\":\"$CASE\",\"timestamp\":\"2026-01-01T00:00:00Z\",\"source_id\":\"$SRC\",
  \"collector\":\"ana\",\"notes\":\"walk\",\"confidence\":0.7,\"entity_ids\":[\"$E\"]}" | jq -r .id)
ITEM=$(curl -s "$B/review?case_id=$CASE" -H "$(hdr rae)" | jq -r ".[]|select(.subject_id==\"$OBS\")|.id")
curl -s $B/review/$ITEM/decision -H "$(hdr rae)" -H 'content-type: application/json' \
  -d '{"decision":"approve"}' | jq .status                            # approved

# 5. a viewer member reads but cannot mutate
curl -s -o /dev/null -w "viewer detail -> %{http_code}\n"  $B/cases/$CASE -H "$(hdr vic)"           # 200
curl -s -o /dev/null -w "viewer intake -> %{http_code}\n" -X POST $B/observations -H "$(hdr vic)" \
  -H 'content-type: application/json' -d "{\"case_id\":\"$CASE\",\"timestamp\":\"2026-01-01T00:00:00Z\",\"source_id\":\"$SRC\",\"collector\":\"vic\",\"entity_ids\":[]}"  # 403

# 6. a partner sees the approved report/export only — never raw material
REP=$(curl -s -X POST $B/cases/$CASE/report -H "$(hdr ana)" | jq -r .id)
curl -s -X POST $B/reports/$REP/publish -H "$(hdr casey)" >/dev/null
curl -s -o /dev/null -w "partner report -> %{http_code}\n"   $B/reports/$REP -H "$(hdr partner)"        # 200
curl -s -o /dev/null -w "partner evidence -> %{http_code}\n" $B/cases/$CASE/evidence -H "$(hdr partner)" # 403

# 7. the unassigned user is refused open / mutate / graph / audit (generic 403)
for p in "" "/graph" "/audit"; do
  curl -s -o /dev/null -w "nomad $p -> %{http_code}\n" $B/cases/$CASE$p -H "$(hdr nomad)"            # 403
done

# 8. membership changes appear in the audit log
curl -s $B/cases/$CASE/audit -H "$(hdr admin)" | jq -r '.[].action' | grep case.member
```

## Frontend

- The **case list** shows only the cases the caller may access (administrators see all);
  the unassigned user gets a calm empty state, not an error.
- The **Case Detail** header shows the viewer's case role ("Your access: …").
- A **Members** tab lists each member's user, global role, case role, status, who
  assigned them and when. Administrators and the case's manager get inline controls to
  assign members (with an explicit case role), change a role, or revoke access; everyone
  else sees the roster read-only.
- 403 reads render the calm "you don't have access" notice rather than an outage.

## Verified guarantees (tests)

`tests/backend/test_case_membership.py` proves: case listing is scoped (admin all,
member assigned-only, unassigned none); the unassigned user is refused open / mutate /
graph / audit with a generic 403 that does not reveal case existence; mutation, review,
and report access are gated by the case role; the review queue shows only items from
cases the caller may review; a partner reaches approved reports only for assigned cases
and never raw material; the graph is scoped; membership add / role-change / deactivate /
reactivate work and are audited; and only an administrator or the case's manager may
manage the roster. The v0.2–v0.5 suites continue to pass (including the PostgreSQL
integration test), confirming the loop still works end to end.
