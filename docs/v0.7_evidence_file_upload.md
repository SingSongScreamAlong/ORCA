# v0.7 — Evidence File Upload + Storage Hardening

v0.7 adds **real manual evidence file upload** on top of the existing Evidence Locker
(v0.3), the content-addressed store, RBAC (v0.4), and case membership (v0.6). An assigned
analyst can upload a lawful file; ORCA hashes it, stores the bytes content-addressed by
SHA-256, records the metadata, and routes it through the same review/quarantine/verify
workflow as any other evidence. Uploads are **case-scoped, hash-verifiable,
access-controlled, auditable, and safe-by-default**.

This milestone is **upload/storage only**. It adds no scraping, dark-web collection,
autonomous hunting, face search, offender/victim targeting, external OSINT integrations,
Palantir sync, AI automation, bulk social monitoring, or direct-contact workflows.

## Safety boundaries (enforced + acknowledged)

Every upload requires an explicit safety acknowledgement (a form field the backend
checks, and a checkbox the UI requires). The boundaries from
[`safety_and_handling.md`](safety_and_handling.md) apply in full:

- No CSAM. No illegally obtained material. No unauthorised private material.
- Nothing for harassment, doxxing, stalking, or vigilante action.
- Urgent or illegal content is reported through authorised channels — ORCA is not a
  reporting authority and is not monitored for emergencies.
- Upload is for lawful, authorised, case-scoped workflows only.

ORCA never executes, decodes, renders, or inspects uploaded content. It hashes and stores
bytes; the rest is metadata.

## Workflow

1. An assigned analyst opens an assigned case and acknowledges the safety boundaries.
2. They upload a lawful file (multipart).
3. The backend reads the bytes (size-capped), computes SHA-256, size, and effective MIME.
4. The upload policy classifies the file (accept / quarantine / reject).
5. An `EvidenceItem` is created, content-addressed, and linked to the case (and an
   observation, if supplied).
6. A reviewer approves / rejects / quarantines it (separation of duties still applies).
7. Authorised users verify the hash and download the bytes.
8. Reports cite only **approved** evidence under **approved** observations.
9. Upload, quarantine, verify, download, and review decisions are all audited.

## Endpoints

| Method & path                              | Purpose                                                |
| ------------------------------------------ | ------------------------------------------------------ |
| `POST /cases/{id}/evidence/upload`         | Multipart upload; hashes, stores, and records the item.|
| `GET  /evidence/{id}/download`             | Download raw bytes — authorised roles only.            |
| `GET  /evidence/{id}`                      | Safe metadata (any reading member of the case).        |
| `POST /evidence/{id}/verify`              | Re-hash stored bytes and report any mismatch.          |
| `POST /evidence/{id}/decision`            | Approve / reject / needs_more_review / quarantine.     |
| `POST /evidence` (metadata-only)           | Preserved from v0.3 (external reference / partner hash).|

The upload is `multipart/form-data`: `file`, `source_id`, `title`, `acknowledged`
(must be true), and optional `evidence_type`, `description`, `observation_id`,
`lawful_basis`, `requires_legal_review`, `sensitive`, `partner_approved`,
`handling_notes`.

## Upload policy (safe-by-default)

Classification happens **before** the bytes are trusted, by filename + declared type:

| Outcome       | When                                                     | Result                                      |
| ------------- | -------------------------------------------------------- | ------------------------------------------- |
| **Reject**    | Extension on the blocked list (executables / scripts).   | `422`; **nothing is stored**.               |
| **Accept**    | Effective MIME on the allow-list.                        | Stored; status `proposed`.                  |
| **Quarantine**| Any other type (unknown / not allow-listed).             | Stored; status `quarantined` pending review.|

Oversize uploads are streamed with a cap and **rejected** (`422`) before anything is
stored. Quarantined and rejected evidence is excluded from reports by construction.

### Configuration

| Setting (`ORCA_…`)                 | Default                                                        |
| ---------------------------------- | ------------------------------------------------------------- |
| `EVIDENCE_MAX_UPLOAD_BYTES`        | `26214400` (25 MiB)                                           |
| `EVIDENCE_ALLOWED_MIME_TYPES`      | pdf, png, jpeg, gif, webp, tiff, plain, csv, markdown, json   |
| `EVIDENCE_BLOCKED_EXTENSIONS`      | `.exe,.dll,.so,.sh,.bat,.ps1,.js,.jar,.msi,…` (executables)   |
| `EVIDENCE_ALLOW_VIEWER_DOWNLOAD`   | `false` (viewers get metadata only)                          |

## Access control (need-to-know + role)

Built on v0.6 case membership. Uploads and downloads are refused with a single generic
403 that never reveals whether a case exists.

| Action                     | admin | case_manager | analyst | reviewer | viewer        | partner_export_viewer | unassigned |
| -------------------------- | :---: | :----------: | :-----: | :------: | :-----------: | :-------------------: | :--------: |
| Upload a file              |   ✓   |      ✓       |    ✓    |    ✓     |               |                       |            |
| Read evidence metadata     |   ✓   |      ✓       |    ✓    |    ✓     |      ✓        |                       |            |
| Download raw bytes         |   ✓   |      ✓       |    ✓    |    ✓     | policy¹       |                       |            |
| Verify hash                |   ✓   |      ✓       |    ✓    |    ✓     |      ✓        |                       |            |

¹ Viewers download **approved** bytes only when `EVIDENCE_ALLOW_VIEWER_DOWNLOAD=true`
(off by default). The `partner_export_viewer` reaches neither raw bytes nor raw metadata —
only approved report packages (v0.6). Everything is additionally scoped to the case:
bytes and metadata for a case you are not assigned to are never reachable.

## Integrity

The stored bytes are content-addressed by their SHA-256, which is the integrity anchor.
The same bytes always produce the same hash (deterministic), and `verify` re-reads the
stored bytes and re-hashes them, surfacing any mismatch rather than hiding it. For the
PostgreSQL backend the bytes live in the filesystem content store (`ORCA_EVIDENCE_STORE`);
the in-memory backend holds them in process for development and tests.

## Audit

| Action                | When                                              |
| --------------------- | ------------------------------------------------- |
| `evidence.uploaded`   | A file is accepted or quarantined on upload.      |
| `evidence.quarantined`| An upload is quarantined by policy.               |
| `evidence.downloaded` | Raw bytes are served to an authorised caller.     |
| `evidence.verified`   | A hash is re-verified.                             |
| `evidence.{approve,reject,needs_more_review,quarantine}` | A reviewer decides the item. |

**Denied attempts** are *not* recorded as separate audit rows: a denied request raises
before the unit of work commits, so any speculative audit write would be rolled back
(the in-memory and SQL backends would disagree). The denial is still enforced; only
*successful* sensitive accesses are audited. This is a deliberate, documented choice.

## Frontend

The Evidence tab gains an **Upload a lawful file** form (visible to users who may create
evidence; the backend also requires active mutating membership) with a **mandatory safety
acknowledgement**, source selection, lawful-basis and handling flags, and upload state.
Each locker row shows the filename, MIME type, size, SHA-256, status, who captured it and
when, with **Verify hash** and — for mutating roles — **Download** actions. Raw download
is hidden from viewers and partner export viewers, and 403s render the calm access notice.

## Walkthrough (curl)

```bash
B=http://localhost:8000/api/v1
hdr() { echo "X-ORCA-User: $1"; }
CASE=$(curl -s $B/cases -H "$(hdr admin)" | jq -r '.[0].id')
SRC=$(curl -s $B/sources -H "$(hdr ana)" | jq -r '.[0].id')
echo "lawful evidence" > note.txt

# analyst uploads (must acknowledge); reviewer downloads the bytes
EV=$(curl -s -X POST $B/cases/$CASE/evidence/upload -H "$(hdr ana)" \
  -F "file=@note.txt;type=text/plain" -F "source_id=$SRC" -F "title=Field note" \
  -F "acknowledged=true" | jq -r .id)
curl -s $B/evidence/$EV/download -H "$(hdr rae)"                                  # bytes
curl -s -X POST $B/evidence/$EV/verify -H "$(hdr rae)" | jq .verified            # true

# a viewer is refused the raw bytes; an unassigned user cannot upload at all
curl -s -o /dev/null -w "viewer download -> %{http_code}\n" $B/evidence/$EV/download -H "$(hdr vic)"   # 403
curl -s -o /dev/null -w "nomad upload  -> %{http_code}\n" -X POST $B/cases/$CASE/evidence/upload \
  -H "$(hdr nomad)" -F "file=@note.txt" -F "source_id=$SRC" -F "title=x" -F "acknowledged=true"        # 403

# a dangerous type is refused outright; an unknown type is quarantined
curl -s -o /dev/null -w "exe upload -> %{http_code}\n" -X POST $B/cases/$CASE/evidence/upload \
  -H "$(hdr ana)" -F "file=@note.txt;filename=bad.exe" -F "source_id=$SRC" -F "title=x" -F "acknowledged=true"  # 422
```

## Verified guarantees (tests)

`tests/backend/test_evidence_upload.py` proves: an assigned analyst can upload; unassigned
users and viewers cannot; partners can neither upload nor download; SHA-256 is
deterministic and verifies; metadata and raw download are case- and role-scoped; dangerous
types are rejected and unknown types quarantined; oversize uploads are rejected; approved
uploaded evidence under an approved observation is cited in reports while quarantined
evidence is excluded; and upload/download/verify are audited. The PostgreSQL integration
test exercises the multipart upload → filesystem content store → download → verify path.
The v0.2–v0.6 suites continue to pass.
