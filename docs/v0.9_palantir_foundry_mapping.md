# v0.9 — Palantir Foundry Ontology Mapping

## 1. Purpose

v0.9 produces a **Foundry-ready ontology specification** and a **local mapping module**
that describe how ORCA's domain model translates into Palantir Foundry object types, link
types, action types, permission rules, and an OSDK-ready application plan. It lets a future
deployment stand ORCA up on a real Foundry tenant without re-deriving the model — while
keeping every ORCA invariant intact.

The mapping lives in `backend/app/foundry_mapping/` and exports JSON via
`python -m app.foundry_mapping.export` to `foundry/`.

## 2. Scope and non-goals

> **Follow-on (v1.1).** This milestone is mapping/spec only. The **v1.1 Foundry Connection
> Spike** ([`v1.1_foundry_connection_spike.md`](v1.1_foundry_connection_spike.md)) adds
> read-only connection *scaffolding* (config, client abstraction, mock client, health
> check) — still no live sync, production writes, or AIP execution.

**In scope:** typed mapping definitions (objects, links, actions, permissions, field
mappings); a deterministic JSON export; validation tests; and this documentation.

**Explicitly out of scope (non-goals):** live Palantir sync, external Palantir API calls,
production Foundry writes, AIP automation, scraping, dark-web collection, autonomous
hunting, face search, offender/victim targeting, external OSINT integrations, bulk social
monitoring, and direct-contact workflows. This milestone performs **no** network I/O — it
only writes local JSON. The forbidden behaviours are enumerated in
`app.foundry_mapping.actions.FORBIDDEN_WORKFLOWS` and a test asserts none is ever expressed
as an action type.

## 3. ORCA → Foundry object type mapping

Thirteen ORCA models map to Foundry object types (full property tables in
[`foundry_ontology_mapping.md`](foundry_ontology_mapping.md)):

| Foundry object type | ORCA source |
| ------------------- | ----------- |
| `OrcaCase` | `CaseRead` |
| `OrcaUser` | `UserRead` |
| `OrcaCaseMembership` | `CaseMemberRead` |
| `OrcaSource` | `SourceRead` |
| `OrcaObservation` | `ObservationRead` |
| `OrcaEvidenceItem` | `EvidenceItemRead` |
| `OrcaEntity` | `EntityRead` |
| `OrcaRelationship` | `RelationshipRead` |
| `OrcaReviewDecision` | `ReviewItemRead` (decided) |
| `OrcaTask` | `ReviewItemRead` (open queue item) |
| `OrcaReport` | `ReportRead` |
| `OrcaReportPackage` | `ReportPackageRead` |
| `OrcaAuditEvent` | `AuditEntry` |

Each property records its ORCA source field and a **mapping kind**: `direct` (a stored
ORCA field), `derived` (computed by ORCA — e.g. evidence `verificationStatus`, case
`handlingLevel`), or `planned` (a target Foundry property not yet in the ORCA model — e.g.
case `priority`, `partnerAgency`). Sensitive properties (`legalFlags`, `handlingNotes`,
`storageUri`, observation/evidence free text, entity `value`) are flagged so the
permission layer can keep them out of partner exports.

## 4. ORCA → Foundry link type mapping

Nineteen link types mirror relationships ORCA already enforces in its relational record
(full table in [`foundry_ontology_mapping.md`](foundry_ontology_mapping.md)): the
`caseContains*` containment links, `caseHasCaseMembership` (the need-to-know roster),
`sourceSupportsObservation`, `observationCitesEvidenceItem`,
`observationSupportsRelationship` (approved observations only),
`entityParticipatesInRelationship`, `reportCitesObservation` (approved only),
`reportPackageIncludesReport`, `reportPackageIncludesEvidenceManifest` (metadata + hashes,
no raw bytes), the three `reviewDecisionAppliesTo*` links, and
`auditEventRecordsActionOnObject`. No link grants a capability ORCA does not already have.

## 5. ORCA → Foundry action type mapping

Twenty action types mirror existing, audited ORCA workflows, each carrying the same RBAC +
case-role gate (full table in [`foundry_action_types.md`](foundry_action_types.md)): case
create/close/archive and membership assign/revoke; source/observation/evidence creation
and evidence upload + hash verify; the review decisions (approve / reject /
needs_more_review for observations and evidence, plus quarantine); relationship creation
from approved observations; report draft and report **package** generation; and report
package download. Every action names the ORCA invariant it preserves; none is autonomous.

## 6. Permission and case membership mapping

Permission rules are **derived from ORCA's real authorization predicates**
(`app.core.rbac` + the case-role helpers) so the exported spec cannot drift from what the
system enforces. A Foundry tenant implements them as object/property-level security and
action gates (details in [`foundry_permission_mapping.md`](foundry_permission_mapping.md)):

- **Need-to-know:** a non-admin must hold an active `OrcaCaseMembership` to see or act on
  a case; `caseHasCaseMembership` is the security pivot.
- **Case role governs:** read material, mutate, review, manage members, view audit, and
  raw-evidence access follow the case role, not just the global role.
- **Raw evidence** (`storageUri` / byte download) is restricted to mutating roles
  (case_manager / analyst / reviewer) and admins.
- **Partner export viewer:** approved report packages only — `can_read_material`,
  `can_access_raw_evidence`, and `can_view_audit` are all false.
- **Admin** is the only role that may override self-review (audited); **unassigned** users
  get a generic denial that does not reveal case existence.

## 7. Audit mapping

`OrcaAuditEvent` (from the append-only `AuditEntry`) carries `actorId`, `action`,
`targetType`, `targetId`, `caseId`, `context`, and `createdAt`. The
`auditEventRecordsActionOnObject` link ties each event to the object it concerns. In
Foundry these become immutable event objects — never updated or deleted — preserving
ORCA's append-only audit guarantee. Every action type sets `writes_audit = true` for the
consequential operations.

## 8. Evidence and hash metadata mapping

`OrcaEvidenceItem` carries the full integrity + provenance set: `sha256` (integrity
anchor), `sizeBytes`, `mimeType`, `originalFilename`, `sourceId` (provenance),
`capturedAt` / `capturedBy`, `accessMethod`, `legalFlags`, `handlingNotes`, `status`, and a
**derived** `verificationStatus` (re-hash of stored bytes →
verified / mismatch / unverified_no_bytes / no_hash). `storageUri` is a content-addressed
pointer and is flagged sensitive — **raw bytes are never bundled into exports**.

## 9. Report package mapping

`OrcaReportPackage` is the partner-ready export object: `reportSha256`, `manifestSha256`,
`handlingLevel`, `generatedBy`, status, and a derived `citedEvidenceCount`. It links to its
report (`reportPackageIncludesReport`) and to the approved evidence it references by
metadata + hash (`reportPackageIncludesEvidenceManifest`). The package is built from
**approved material only**; proposed/rejected/needs_more_review/quarantined items and raw
files are excluded — exactly as v0.8 enforces.

## 10. OSDK application plan

See [`foundry_osdk_app_plan.md`](foundry_osdk_app_plan.md). In summary: generate a
TypeScript OSDK from the published ontology; build read views (case workspace, review
queue, graph, report packages) bound to OSDK object sets with object/property security; and
wire analyst actions to the Foundry action types — preserving "AI proposes, analysts
decide". The existing ORCA Next.js app is the reference UI; an OSDK build would swap the
data layer, not the workflow.

## 11. AIP use cases (propose-only)

AIP is permitted only in an **assistive, propose-only** capacity, never autonomous:
candidate relationship suggestions from co-occurrence, candidate entity-merge
suggestions, draft report summarisation of **approved** material, and review-queue triage
hints. Every AIP output is a *proposal* that enters the human review queue — it never
approves, never writes a finding about a person, and never bypasses audit. No AIP
automation, scheduling, or autonomous action is in scope (`aip_automation` is a forbidden
workflow).

> **Realised in v1.0.** The Analyst Copilot
> ([`v1.0_aip_assisted_analyst_copilot.md`](v1.0_aip_assisted_analyst_copilot.md))
> implements exactly this propose-only contract locally, behind an `AiProvider` seam that an
> AIP-backed provider would slot into without changing any boundary.

## 12. Data safety boundaries

The mapping preserves ORCA's boundaries (`docs/safety_and_handling.md`): lawful,
analyst-controlled, human-reviewed; no CSAM storage/handling; no scraping or autonomous
collection; need-to-know access; partner isolation; append-only audit. The forbidden
workflows in §2 are enforced by tests, not just prose.

## 13. Migration / sync strategy

A real-tenant rollout would be **one-directional and explicit**, never the live sync this
milestone forbids: (1) publish the ontology from the exported spec; (2) backfill object
sets from ORCA's PostgreSQL system of record via a vetted, audited batch export; (3) bind
the OSDK app's actions to ORCA's API (ORCA stays the system of record and the authorization
authority); (4) reconcile on a schedule with full audit. Any write path would route through
ORCA's services so RBAC, case membership, separation of duties, and audit still apply.

## 14. Open questions for a real Foundry tenant

- Mapping ORCA's per-case need-to-know onto Foundry's security model (Markings / Restricted
  Views vs. application-level checks) — which enforces the generic-403 non-leakage best?
- Where is the system of record — ORCA's PostgreSQL, or Foundry? (This spec assumes ORCA.)
- How are `planned` properties (priority, partnerAgency, assignedTo, source capture fields)
  finalised before publishing?
- Foundry representation of the content-addressed evidence store and hash verification.
- Tenant-specific handling/marking taxonomy for `legalFlags` and `handlingLevel`.
- Retention/deletion: reconciling ORCA's "delete case never deletes evidence" with Foundry.

## Export & validation

```bash
cd backend
python -m app.foundry_mapping.export        # writes foundry/*.json (deterministic)
python -m pytest ../tests/backend/test_foundry_mapping.py -q
```

`tests/backend/test_foundry_mapping.py` proves: every required object/link/action is
mapped; no forbidden workflow is expressed; actions carry real capability + case-role
gates and preserve a named invariant; partner export viewers get no raw material; viewers
get no raw bytes; reports/packages map approved material only; evidence carries SHA-256 +
provenance + verification; audit events are readable objects; and the committed
`foundry/*.json` is in sync and deterministic.
