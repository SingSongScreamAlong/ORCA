# v1.1 — Foundry Connection Spike

## Purpose

ORCA is *Palantir-ready* (v0.9 mapping/spec) but not *Palantir-connected*. v1.1 takes the
**smallest safe step** toward connected: a configuration shape, a read-only client
abstraction, a deterministic mock client, honest real-client scaffolding, and a secret-free
health check. It is a **spike, not a full integration**.

## What this is — and is not

- **Is:** connection *scaffolding*. Config that defaults to disabled; a read-only
  `FoundryClient` abstraction; a `MockFoundryClient` for deterministic tests; a real-client
  placeholder that fails gracefully; a health endpoint + CLI.
- **Is not:** full Foundry sync, production writes, live AIP execution, or any data
  movement. v0.9 was **mapping/spec only**; v1.1 adds connection scaffolding only.

## Safety properties

- **Read-only by default**, and disabled by default (`ORCA_FOUNDRY_ENABLED=false`).
- **No credentials required** to run the app or the tests — tests use the mock client.
- **No secrets committed.** Credentials come from environment variables; `.env` is
  gitignored; `.env.example` carries placeholders only.
- **Secrets are never logged.** `FoundryConfig` redacts `client_secret`/`token` in `repr`,
  `str`, and `safe_dict()`; the health output and error messages contain no secret values.
- **No real, sensitive, or case data** is used; the mock returns only synthetic shapes.
- **No raw evidence** is transmitted anywhere.

## Module layout (`backend/app/foundry/`)

| File | Role |
| ---- | ---- |
| `config.py` | `FoundryConfig` — reads `ORCA_FOUNDRY_*`, disabled by default, validates only when enabled, redacts secrets. |
| `client.py` | `FoundryClient` protocol (read-only) + `build_foundry_client()`. |
| `mock_client.py` | `MockFoundryClient` — deterministic synthetic data (default for dev/tests). |
| `real_client.py` | `RealFoundryClient` — honest placeholder; probes for an SDK, fails gracefully with guidance. |
| `health.py` | `foundry_health()` + `python -m app.foundry.health` CLI. |
| `errors.py` | `FoundryError` and friends (never carry secrets). |

## Configuration

| Variable | Meaning |
| -------- | ------- |
| `ORCA_FOUNDRY_ENABLED` | `false` (default) disables everything; `true` enables the real client path. |
| `ORCA_FOUNDRY_TENANT_URL` | Tenant base URL (e.g. `https://your-tenant.palantirfoundry.com`). |
| `ORCA_FOUNDRY_TOKEN` | A pre-issued bearer token (auth method `token`). **Secret.** |
| `ORCA_FOUNDRY_CLIENT_ID` / `ORCA_FOUNDRY_CLIENT_SECRET` | OAuth2 client-credentials (auth method `client_credentials`). **Secret.** |
| `ORCA_FOUNDRY_ONTOLOGY_API_NAME` | The published ontology's API name. |
| `ORCA_FOUNDRY_TEST_OBJECT_TYPE` / `ORCA_FOUNDRY_TEST_OBJECT_ID` | A single harmless demo object to read during a manual test. |
| `ORCA_FOUNDRY_SDK_MODULE` | Optional: the Python module the real client probes for (default `foundry_sdk`). |

Provide **either** a token **or** a client id + secret. When enabled, `tenant_url`,
`ontology_api_name`, and one auth method are required; missing fields produce a clear
validation error.

## Read-only client surface

`health_check()`, `get_object_type_metadata(object_type)`,
`get_object_by_id(object_type, object_id)`, `list_demo_objects(object_type, limit)`. No
write methods exist in this milestone.

## Health check

- **API:** `GET /api/v1/integrations/foundry/health` — **admin-only**, read-only.
- **CLI:** `python -m app.foundry.health`.

Reports `enabled`, `configured`, `auth_method`, `host` (host only), `ontology_api_name`,
`mode` (`disabled` / `mock` / `real`), and `ok` + `message` (and `errors` when
misconfigured). It never emits secrets. Example (disabled):

```json
{ "enabled": false, "configured": false, "mode": "disabled", "ok": null,
  "message": "Foundry integration is disabled.", "token": null }
```

## Audit

A Foundry health check is a **connection/admin diagnostic, not a case action**, so it is
**not** written to the case audit log. (If real read calls are added later, access to
sensitive objects should be audited at that point.)

## Manual test against a real tenant

See [`foundry_connection_setup.md`](foundry_connection_setup.md) for the full procedure.
In short: set the `ORCA_FOUNDRY_*` env vars in a local (gitignored) `.env`, then run
`python -m app.foundry.health`. With no real SDK wired in, the real client fails gracefully
with guidance — which is the honest current state of the spike.

## Tests

`tests/backend/test_foundry_connection.py` proves: config defaults disabled; missing
credentials don't break startup; enabling without config gives a clear validation error;
secrets are redacted; the mock client's read-only methods work; the health endpoint/CLI is
safe and admin-only; and the real client fails gracefully (and secret-free) without its
SDK. **No test requires a live Foundry tenant.** The v0.2–v1.0 suites continue to pass.

## Open questions for a real OSDK/Foundry setup

- The exact official Python OSDK/client package name and its read API (this spike does not
  guess it — the real client is a clearly-marked placeholder).
- Auth flow specifics for the chosen method (token vs. OAuth2 client credentials) and token
  lifetime/refresh.
- How ORCA's per-case need-to-know and non-leakage map onto Foundry's security model.
- Which harmless demo object type/id to read first in a tenant.
