# Foundry Permission Mapping

Reference for the ORCA → Foundry **permission** mapping. Authoritative form:
`foundry/permissions.json` (generated by `python -m app.foundry_mapping.export`). The
rules are **derived from ORCA's real authorization predicates** in `app.core.rbac` and the
case-role helpers, so the exported spec cannot drift from what ORCA actually enforces.

## Principle

RBAC answers *what kind of user are you?*; case membership answers *which case may you
touch, and how?*. A Foundry tenant implements both as a combination of action gates and
object/property-level security, with `caseHasCaseMembership` as the need-to-know pivot.

## Case-role permission matrix

| Case role | read material | mutate | review | manage members | view audit | raw evidence | approved reports / export |
| --------- | :-----------: | :----: | :----: | :------------: | :--------: | :----------: | :-----------------------: |
| `case_manager` | ✓ | ✓ | | ✓ | ✓ | ✓ | ✓ |
| `analyst` | ✓ | ✓ | | | | ✓ | ✓ |
| `reviewer` | ✓ | ✓ | ✓ | | ✓ | ✓ | ✓ |
| `viewer` | ✓ | | | | | | ✓ |
| `partner_export_viewer` | | | | | | | ✓ |

(Reviewers may also propose; separation of duties forbids approving one's own proposal
without an audited admin override.)

## Global principals

- **admin** — superuser: accesses and manages every case regardless of membership, and is
  the only role that may override the self-review control (audited).
- **unassigned** — a user with no active membership can do nothing with a case, including
  learning whether it exists; ORCA returns a generic 403 and Foundry security must not leak
  object existence either.

## Raw evidence & partner isolation

- **Raw evidence bytes** (`OrcaEvidenceItem.storageUri` / byte download) are restricted to
  admins and mutating roles (case_manager / analyst / reviewer). Viewers see metadata only.
- **`partner_export_viewer`** reaches **only** approved `OrcaReportPackage` objects for
  assigned cases — never raw observations, evidence, the relationship graph, or the audit
  log. In the export, the partner row has `can_read_material`, `can_access_raw_evidence`,
  and `can_view_audit` all false; a test enforces this.

## Property-level security

Properties flagged `sensitive` in the object mapping (`legalFlags`, `handlingNotes`,
`storageUri`, observation/evidence free text, entity `value`, report `body`) should carry
property-level restrictions so they are excluded from partner-facing object sets and
exports, mirroring how ORCA's report packages reference evidence by metadata + SHA-256
rather than raw content.
